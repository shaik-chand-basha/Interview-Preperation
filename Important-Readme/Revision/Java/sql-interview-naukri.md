What are the different types of normalizations?</strong></span></h3><p><span style="background-color:transparent;color:#000000;">In database design, several types of normalization are used to reduce data redundancy, improve data integrity, and ensure efficient data retrieval. The different types of normalizations are&nbsp;</span></p><ul><li><span style="background-color:transparent;color:#000000;"><strong>First Normal Form (1NF):</strong> This normalization ensures that the data in each table's column is atomic, meaning that it cannot be further broken down into smaller pieces. It also eliminates duplicate rows from the table.</span><br>&nbsp;</li><li><span style="background-color:transparent;color:#000000;"><strong>Second Normal Form (2NF):&nbsp;</strong>This normalization eliminates partial dependencies by ensuring that each non-key column in a table is dependent on the entire primary key rather than on its part of it.</span><br>&nbsp;</li><li><span style="background-color:transparent;color:#000000;"><strong>Third Normal Form (3NF):</strong> This normalization eliminates transitive dependencies by ensuring that each non-key column in a table is dependent only on the primary key and not on any other non-key columns.</span><br>&nbsp;</li><li><span style="background-color:transparent;color:#000000;"><strong>Boyce-Codd Normal Form (BCNF):&nbsp;</strong>This normalization is an extension of 3NF and ensures that each determinant in a table is a candidate key.</span><br>&nbsp;</li><li><span style="background-color:transparent;color:#000000;"><strong>Fourth Normal Form (4NF):&nbsp;</strong>This normalization eliminates multivalued dependencies by ensuring that each non-key column in a table is dependent on the entire primary key and not on any subsets.</span><br>&nbsp;</li><li><span style="background-color:transparent;color:#000000;"><strong>Fifth Normal Form (5NF):&nbsp;</strong>This normalization is known as the Project-Join Normal Form and ensures that each table in a database has a single theme or topic.</span><br>&nbsp;</li></ul><p><span style="background-color:transparent;color:#000000;">The normalization process is iterative, and it may be necessary to apply multiple normalizations to achieve the desired data organization and efficiency level.</span></p><h3 id="3.7."><span style="background-color:transparent;color:#000000;"><strong>32. Explain Boyce-Codd Normal Form (BCNF).</strong></span></h3><p><span style="background-color:transparent;color:#000000;">Boyce-Codd Normal Form (BCNF) is a normalization technique used in database design to eliminate redundancy and improve data integrity. In BCNF, each determinant (i.e., attribute or set of attributes that uniquely determine another attribute) in a table must be a candidate key (i.e., a unique identifier for each row).</span></p><p><span style="background-color:transparent;color:#000000;">In simpler terms, BCNF ensures that each attribute in a table depends on the entire primary key rather than on just a part of it. This helps to reduce data redundancy and improve data integrity by ensuring that each piece of data in the table is only stored once and can be accessed efficiently.</span></p><p><span style="background-color:transparent;color:#000000;">BCNF is an extension of the Third Normal Form (3NF) and is helpful in situations where 3NF is insufficient to eliminate all forms of redundancy in a table.&nbsp;</span></p><p><span style="background-color:transparent;color:#000000;">However, it is essential to note that achieving BCNF can sometimes result in a higher number of tables and more complex relationships between them. Unlike a regular view, which is a virtual table that is generated dynamically upon query execution, a materialized view stores data to improve performance, especially for complex queries.</p><p>Example:</p><pre><code class="language-plaintext">CREATE MATERIALIZED VIEW emp_dept_view AS
SELECT e.first_name, d.department_name
FROM employees e
JOIN departments d ON e.department_id = d.department_id;
</code></pre><p>Materialized views are useful for improving performance but require periodic refreshing to keep the data current.</p><h3 id="4.20."><strong>70. What is the difference between EXISTS and IN in SQL?</strong></h3><p>Both <strong>EXISTS</strong> and <strong>IN</strong> are used to filter data based on subqueries, but they operate differently:</p><ul><li><strong>EXISTS</strong>: Checks if the subquery returns any rows. It is efficient when the subquery result set is large.</li><li><strong>IN</strong>: Compares values from the outer query with a list of values returned by the subquery. It performs better when the subquery returns a smaller result set.</li></ul><p>Example:</p><pre><code class="language-plaintext">-- Using EXISTS
SELECT e.first_name
FROM employees e
WHERE EXISTS (SELECT 1 FROM departments d WHERE e.department_id = d.department_id);

-- Using IN
SELECT e.first_name
FROM employees e
WHERE e.department_id IN (SELECT department_id FROM departments);
</code></pre><h3 id="4.21."><strong>71. What is Database Normalization and why is it important?</strong></h3><p><strong>Database Normalization</strong> is the process of organizing a database to reduce redundancy and improve data integrity. It involves dividing large tables into smaller, related tables and defining relationships between them. The key goals of normalization are:</p><ol><li>Eliminate redundancy.</li><li>Ensure data consistency.</li><li>Facilitate easier maintenance.</li></ol><p>The main normal forms include:</p><ul><li><strong>1NF</strong>: Ensures that each column contains atomic values.</li><li><strong>2NF</strong>: Requires 1NF compliance and removes partial dependencies.</li><li><strong>3NF</strong>: Requires 2NF compliance and eliminates transitive dependencies.</li></ul><p>Normalization optimizes database structure but may introduce performance issues with excessive joins.</p><h3 id="4.22."><strong>72. What are Stored Procedures and why are they useful?</strong></h3><p>A <strong>Stored Procedure</strong> is a compiled group of SQL statements that can be executed as a single unit. Stored procedures are useful for:</p><ol><li><strong>Code reusability</strong>.</li><li><strong>Improved performance</strong>, as they are precompiled.</li><li><strong>Security</strong> by controlling access to data.</li></ol><p>Example:</p><pre><code class="language-plaintext">CREATE PROCEDURE UpdateSalary(IN emp_id INT, IN new_salary DECIMAL)
BEGIN
    UPDATE employees SET salary = new_salary WHERE employee_id = emp_id;
END;
</code></pre><p>Stored procedures can accept input parameters and return output values, making them a powerful tool for automating tasks in SQL.</p><h3 id="4.23."><strong>73. Explain the concept of Database Denormalization.</strong></h3><p><strong>Denormalization</strong> is the process of combining tables or adding redundant data to a database to improve query performance. While normalization reduces redundancy, denormalization may reintroduce it to reduce the complexity of joins and speed up read queries.</p><p>Example: Instead of keeping employee and department data in separate tables, denormalization might store department information directly within the employee table.</p><p>Denormalization trades data redundancy for query performance, particularly in systems where read performance is more critical than storage efficiency.</p><h3 id="4.24."><strong>74. What is the difference between Clustered and Non-Clustered Indexes?</strong></h3><ul><li><strong>Clustered Index</strong>: Determines the physical order of data in a table. A table can only have one clustered index, and the data is stored based on this index.</li><li><strong>Non-Clustered Index</strong>: Does not affect the physical order of the data. A table can have multiple non-clustered indexes, and these indexes contain pointers to the actual data.</li></ul><p>Example:</p><pre><code class="language-plaintext">CREATE CLUSTERED INDEX idx_employee_id ON employees(employee_id);
CREATE NONCLUSTERED INDEX idx_employee_name ON employees(first_name, last_name);
</code></pre><p>Clustered indexes improve the performance of range queries, while non-clustered indexes optimize individual lookups.</p><h3 id="4.25."><strong>75. How do you handle Deadlocks in SQL?</strong></h3><p>A <strong>deadlock</strong> occurs when two or more transactions hold locks on resources and each is waiting for the other to release the lock, causing a cycle. To handle deadlocks:</p><ol><li><strong>Minimize locking time</strong> by keeping transactions short.</li><li><strong>Use consistent locking order</strong> in your application.</li><li><strong>Set proper isolation levels</strong> to avoid unnecessary locking.</li></ol><p>SQL Server automatically detects deadlocks and resolves them by rolling back one of the transactions. You can also manually resolve deadlocks by catching the error and retrying the transaction.</p><h3 id="4.26."><strong>76. What is the purpose of the EXPLAIN command in SQL?</strong></h3><p>The <strong>EXPLAIN</strong> command is used to analyze and display the execution plan of a SQL query. It shows how the query optimizer plans to execute the query, including details like table scans, index usage, join operations, and cost estimates.</p><p>Example:</p><pre><code class="language-plaintext">EXPLAIN SELECT * FROM employees WHERE salary &gt; 50000;
</code></pre><p>By reviewing the execution plan, you can identify performance bottlenecks, such as full table scans, and take steps to optimize the query.</p><h3 id="4.27."><strong>77. What are SQL Subqueries, and how are they used?</strong></h3><p>A <strong>subquery</strong> is a query nested inside another query. Subqueries can be used to return results that are used by the outer query. Subqueries can appear in SELECT, FROM, WHERE, or HAVING clauses.</p><p>Example:</p><pre><code class="language-plaintext">SELECT first_name
FROM employees
WHERE salary &gt; (SELECT AVG(salary) FROM employees);
</code></pre><p>This query returns employees whose salary is higher than the average salary, which is calculated by the subquery.</p><h3 id="4.28."><strong>78. What are SQL Transactions and their properties?</strong></h3><p>A <strong>Transaction</strong> in SQL is a sequence of operations performed as a single logical unit. It ensures data integrity through <strong>ACID properties</strong>:</p><ol><li><strong>Atomicity</strong>: Ensures all operations in a transaction are completed or none are.<br>&nbsp;</li><li><strong>Consistency</strong>: Guarantees that the database remains in a consistent state after the transaction.<br>&nbsp;</li><li><strong>Isolation</strong>: Prevents concurrent transactions from interfering with each other.<br>&nbsp;</li><li><strong>Durability</strong>: Ensures that once a transaction is committed, it remains so, even in the event of a system failure.</li></ol><p>Example:</p><pre><code class="language-plaintext">BEGIN TRANSACTION;
UPDATE employees SET salary = salary * 1.1 WHERE department_id = 1;
COMMIT;
</code></pre><p><br>If any part of the transaction fails, you can use ROLLBACK to revert all changes.</p><h3 id="4.29."><strong>79. How do you create and use a Pivot Table in SQL?</strong></h3><p>A <strong>Pivot Table</strong> in SQL allows you to transform data from rows into columns. This is useful for generating cross-tab reports.</p><p>Example:</p><pre><code class="language-plaintext">SELECT department,
    SUM(CASE WHEN gender = 'Male' THEN 1 ELSE 0 END) AS male_count,
    SUM(CASE WHEN gender = 'Female' THEN 1 ELSE 0 END) AS female_count
FROM employees
GROUP BY department;
</code></pre><p><br>This query counts the number of males and females in each department, pivoting the gender data into separate columns.</p><h3 id="4.30."><strong>80. What is the difference between DELETE and TRUNCATE in SQL?</strong></h3><ul><li><strong>DELETE</strong>: Removes rows from a table one by one, and each row deletion is logged. It can be rolled back using a transaction.</li><li><strong>TRUNCATE</strong>: Removes all rows from a table quickly by deallocating the data pages. It is faster but cannot be rolled back in most databases.</li></ul><p>Example:</p><pre><code class="language-plaintext">DELETE FROM employees WHERE employee_id = 1001;
TRUNCATE TABLE employees;</code></pre><p><br>DELETE is more flexible, while TRUNCATE is faster for large datasets.</p><div class="parent-heading"><h2 id="5."><strong>PostgreSQL Interview Questions</strong></h2><div class="line" style="width:100%;margin-top:50px;"></div></div><h3 id="5.1."><strong>81. What is PostgreSQL?</strong></h3><p><strong>PostgreSQL</strong> is an open-source, object-relational database management system (ORDBMS) that emphasizes extensibility and compliance with SQL standards. It supports both SQL and JSON for relational and non-relational queries. PostgreSQL is widely known for its robustness, performance, and advanced features like complex queries, foreign keys, triggers, updatable views, and transaction integrity.</p><h3 id="5.2."><strong>82. What is the difference between PostgreSQL and MySQL?</strong></h3><ul><li><strong>PostgreSQL</strong> is an object-relational database management system (ORDBMS), whereas <strong>MySQL</strong> is a purely relational database management system (RDBMS).<br>&nbsp;</li><li>PostgreSQL supports advanced features like custom data types, table inheritance, and function overloading, making it highly extensible.<br>&nbsp;</li><li>MySQL has a simpler system architecture and is more focused on read-heavy applications, while PostgreSQL is designed for complex, write-heavy operations.<br>&nbsp;</li><li>PostgreSQL follows SQL standards more strictly than MySQL.</li></ul><h3 id="5.3."><strong>83. How do you create a new database in PostgreSQL?</strong></h3><p>To create a new database in PostgreSQL, you use the CREATE DATABASE command. For example:</p><pre><code class="language-plaintext">CREATE DATABASE testdb;
</code></pre><p><br>This command will create a new database named testdb. You can connect to this database using the \c testdb command in the PostgreSQL command-line interface.</p><h3 id="5.4."><strong>84. What is a CTE (Common Table Expression) in PostgreSQL?</strong></h3><p>A <strong>Common Table Expression (CTE)</strong> in PostgreSQL is a temporary result set defined within a SQL query. It is typically used to make queries more readable by breaking them into smaller parts. CTEs are created using the WITH clause. Example:</p><pre><code class="language-plaintext">WITH sales_summary AS (
    SELECT department, SUM(sales) AS total_sales
    FROM sales
    GROUP BY department
)
SELECT * FROM sales_summary WHERE total_sales &gt; 10000;
</code></pre><h3 id="5.5."><strong>85. How does PostgreSQL handle indexing?</strong></h3><p>PostgreSQL supports various types of indexes, including <strong>B-tree</strong>, <strong>Hash</strong>, <strong>GIN</strong> (Generalized Inverted Index), and <strong>GiST</strong> (Generalized Search Tree). Indexes in PostgreSQL are automatically used by the query planner to optimize query performance. They help in speeding up the retrieval of rows from a database at the cost of additional storage space and slower inserts/updates. Example of creating an index:</p><pre><code class="language-plaintext">CREATE INDEX idx_employee_id ON employees(employee_id);
</code></pre><h3 id="5.6."><strong>86. What are TOAST tables in PostgreSQL?</strong></h3><p><strong>TOAST</strong> (The Oversized-Attribute Storage Technique) is a mechanism in PostgreSQL for storing large data, such as big text or binary objects. PostgreSQL stores data in tuples, and if a tuple exceeds a certain size limit (usually 8KB), TOAST will automatically compress and store it in an external storage space to avoid performance issues.</p><h3 id="5.7."><strong>87. What are the advantages of using JSONB in PostgreSQL?</strong></h3><p><strong>JSONB</strong> is a binary representation of JSON data in PostgreSQL. It is optimized for efficient storage and faster query performance when working with JSON data. The advantages of JSONB over plain JSON include:</p><ul><li>Faster access and retrieval of JSON data.<br>&nbsp;</li><li>JSONB is stored in a decomposed binary format, allowing efficient indexing.<br>&nbsp;</li><li>Supports operators like @&gt;, &lt;@, and ? for advanced querying of JSON data.</li></ul><h3 id="5.8."><strong>88. How do you perform replication in PostgreSQL?</strong></h3><p>PostgreSQL supports multiple replication methods, including <strong>Streaming Replication</strong>, <strong>Logical Replication</strong>, and <strong>Synchronous Replication</strong>.</p><ul><li><strong>Streaming Replication</strong>: Sends real-time changes from the primary server to a standby server.<br>&nbsp;</li><li><strong>Logical Replication</strong>: Allows selective data replication (row-level) between databases.<br>&nbsp;</li><li><strong>Synchronous Replication</strong>: Ensures that each transaction is confirmed on both primary and standby servers before committing.<br>&nbsp;</li></ul><p>Example of setting up streaming replication:</p><ul><li>Configure the primary server to enable wal_level to replica.</li><li>Set up a standby server and configure the recovery.conf file.</li></ul><h3 id="5.9."><strong>89. How can you perform full-text search in PostgreSQL?</strong></h3><p>PostgreSQL offers built-in support for <strong>full-text search</strong> through the tsvector and tsquery data types. Full-text search is used to search for words and phrases within text fields, enabling more complex querying options. Example:</p><pre><code class="language-plaintext">SELECT * FROM documents
WHERE to_tsvector('english', content) @@ to_tsquery('PostgreSQL &amp; Database');
</code></pre><p><br>This query searches for documents containing both "PostgreSQL" and "Database" in their content.</p><h3 id="5.10."><span style="color:hsl(0,0%,0%);"><strong>90. Without periodic vacuuming, PostgreSQL tables would bloat, leading to reduced performance.&nbsp;</span></p><p><strong>Example:</strong></p><pre><code class="language-plaintext">VACUUM FULL employees;
</code></pre><p><br>This command reclaims storage and optimizes the performance of the employees table.</p><div class="parent-heading"><h2 id="6."><span style="color:hsl(0,0%,0%);"><strong>SQL MCQs</strong></span></h2><div class="line" style="width:100%;margin-top:50px;"></div></div><h3 id="6.1."><span style="color:hsl(0,0%,0%);"><strong>91.
