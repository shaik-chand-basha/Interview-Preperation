<h3 dir="ltr" id="1.-what-is-a-git-repository?-agitr">1. What is a Git repository?</h3>
<p dir="ltr" class="p-margin">A Git repository stores a project's files and revision history and facilitates version control by tracking changes made over time. It can be located locally within a folder on your device or an online platform like GitHub. This enables users to collaborate, revert to previous versions, and efficiently manage project development using commands like commit, push, and pull.</p>
<h3 dir="ltr" id="2.-how-does-git-work?-gitop">2. How does Git work?</h3>
<p dir="ltr" class="p-margin">Git operates by recording changes made to files and directories in a project, capturing snapshots of its evolving condition. Users can oversee alterations, create branches for simultaneous development, merge branches, and revert to previous states if required. It also promotes collaboration and ensures effective version control in software development endeavors.</p>
<h3 dir="ltr" id="3.-what-is-git-add?-the&lt;c">3. What is git add?</h3>
<p dir="ltr" class="p-margin">The <code>git add</code> command is used in Git to stage changes for inclusion in the next commit. It prepares modifications, additions, or deletions made to files in the working directory, marking them to be included in the upcoming commit snapshot. Note this command does not actually commit the changes but prepares them for staging.</p>
<h3 dir="ltr" id="4.-what-is-git-push?-the&lt;c">4. What is git push?</h3>
<p dir="ltr" class="p-margin">The <code>git push</code> command is used in Git to upload local repository content to a remote repository. It transfers committed changes from the local repository to a remote one, typically on a server like GitHub or GitLab. This command enables collaboration by allowing users to share their changes with others on the same project.</p>
<p dir="ltr" class="p-margin">You can learn more about <a href="https://www.datacamp.com/tutorial/git-push-pull" target="_blank" rel="noopener">Git push and pull</a> in our separate tutorial.</p>
<h3 dir="ltr" id="5.-what-is-git-status?-the&lt;c">5. What is git status?</h3>
<p dir="ltr" class="p-margin">The <code>git status</code> command displays the current state of the repository in Git. It provides information about which files have been modified, which are staged for the next commit, and which are untracked. It helps users track the progress of their work and identify any changes that need to be committed or staged.</p>
<h3 dir="ltr" id="6.-what-is-a-commit-in-git?-a&lt;ahr">6. What is a commit in Git?</h3>
<p dir="ltr" class="p-margin">A <a href="https://www.datacamp.com/tutorial/git-commit" target="_blank" rel="noopener">commit</a> represents a snapshot of the changes made to files in a repository at a specific point in time. When you commit changes in Git, you are effectively saving the current state of your files and can provide a descriptive message that explains the changes made (which is recommended).</p>
<p dir="ltr" class="p-margin">Each commit creates a unique identifier, allowing you to track the history of changes in the repository. Commits play a crucial role in version control, as they provide a way to revert to previous states of the project, review the history of changes, and collaborate with others by sharing updates.</p>

<h3 dir="ltr" id="7.-what-is-branching-in-git?-branc">7. What is branching in Git?</h3>
<p dir="ltr" class="p-margin">Branching refers to the practice of diverging from the main line of development (typically called the "master" branch) to work on new features, fixes, or experiments without affecting the main codebase. It allows multiple parallel lines of development to coexist within the same repository.</p>
<p dir="ltr" class="p-margin">Each branch represents a separate line of development with its own set of commits, enabling developers to work on different features or fixes simultaneously. Branching facilitates collaboration, experimentation, and organization within a project, as changes made in one branch can be merged back into the main codebase once they are completed and tested.</p>
<h3 dir="ltr" id="8.-what-is-a-conflict-in-git?-confl">8. What is a conflict in Git?</h3>
<p dir="ltr" class="p-margin">Conflicts arise when conflicting changes are made to the same part of a file or files by different contributors, typically during a merge or rebase operation. Git cannot automatically resolve these conflicting changes, requiring manual intervention by the user to resolve the discrepancies.</p>
<p dir="ltr" class="p-margin">Thus, to resolve conflicts, the conflicted files must be reviewed and edited based on the best suited reconciliation before the resolved version is committed.</p>

Namely, a merge is the process of combining the changes from different branches into a single branch, typically the main branch (e.g., master or main).</p>

<h3 dir="ltr" id="10.-what-is-a-remote-in-git?-aremo">10. What is a remote in Git?</h3>
<p dir="ltr" class="p-margin">A remote is a repository hosted on a server or another computer for collaboration and sharing code with others. It serves as a centralized location where developers can push their local changes and pull changes made by others.</p>
<p dir="ltr" class="p-margin">Remotes are typically set up on hosting platforms like GitHub, GitLab, or Bitbucket, and they enable distributed development and facilitate teamwork by providing a common location for storing and synchronizing project code among multiple contributors.</p>
<h3 dir="ltr" id="11.-what-is-the-difference-between-git-fetch-and-git-pull?-thema">11. What is the difference between git fetch and git pull?</h3>
<p dir="ltr" class="p-margin">The main <a href="https://www.datacamp.com/tutorial/git-fetch-vs-pull" target="_blank" rel="noopener">difference between git fetch and git pull</a> lies in what they do and how they update the local repository.</p>
<p dir="ltr" class="p-margin">The <code>git fetch</code> command retrieves changes from a remote repository to the local repository. It updates the remote-tracking branches (e.g., origin/master) in the local repository to reflect the state of the remote repository, but it does not update the working directory or merge any changes into the current branch. This means that after fetching, you can review the changes made in the remote repository without affecting your local work.</p>
<p dir="ltr" class="p-margin">The <code>git pull</code> command also retrieves changes from a remote repository, but it goes a step further by fetching changes and merging them into the current branch in one step. It essentially performs a git fetch followed by a git merge to incorporate the changes from the remote repository into the current branch.</p>

<p dir="ltr" class="p-margin">Using <code>git revert</code> creates a new commit that undoes the changes introduced by the original commit, effectively reverting the changes without altering the commit history. This approach is safer than <code>git reset</code> or <code>git amend</code>, which can alter the commit history and cause issues for collaborators who have already pulled the changes.</p>
<h3 dir="ltr" id="13.-what-does-git-reset-do?-the&lt;c">13. What does git reset do?</h3>
<p dir="ltr" class="p-margin">The <code>git reset</code> command resets the current HEAD to a specified state. This means it can be used to undo changes, unstage files, or move the HEAD pointer to a different commit. Note, there are three main modes of <code>git reset</code>:</p>
<ul>
<li><code>--soft</code>: Resets the HEAD pointer to a specific commit, keeping changes staged. Files remain modified in the working directory, allowing you to re-commit them.</li>
</ul>
<ul>
<li><code>--mixed</code>: Resets the HEAD pointer to a specific commit, unstaging changes. Files remain modified in the working directory, but changes are not staged for commit.</li>
</ul>
<ul>
<li><code>--hard</code>: Resets the HEAD pointer to a specific commit, discarding all changes in the working directory and staging area. Use with caution, as it permanently deletes uncommitted changes.</li>
</ul>
<h3 dir="ltr" id="14.-what-is-git-stash?-&lt;code">14. What is git stash?</h3>
<p dir="ltr" class="p-margin"><code>git stash</code> is a Git command that temporarily stores changes in the working directory that are not ready to be committed. It allows developers to save their modifications without committing them to the repository.</p>
<p dir="ltr" class="p-margin">Stashing is useful when switching branches, but you don't want to commit or lose your changes. Later, you can apply the stashed changes to your working directory or pop them off the stash stack to continue working on them.</p>
<h3 dir="ltr" id="15.-what-is-git-reflog?-&lt;ahre">15. What is git reflog?</h3>
<p dir="ltr" class="p-margin"><a href="https://www.datacamp.com/tutorial/git-reflog" target="_blank" rel="noopener">git reflog</a> is a Git command used to view the reference logs, which record changes to the HEAD pointer and the history of commits that have been checked out in the repository. It provides a chronological list of recent actions performed in the repository, including commits, checkouts, merges, and resets.</p>
<p dir="ltr" class="p-margin">The reflog is helpful for recovering lost commits or branches and understanding the sequence of actions taken in the repository.</p>

<p dir="ltr" class="p-margin">Handling large files in Git can be challenging due to their impact on repository size and performance. <h3 dir="ltr" id="19.-what-is-the-use-of-git-submodule-and-how-do-you-update-one?-the&lt;c">19. What is the use of git submodule and how do you update one?</h3>
<p dir="ltr" class="p-margin">The <code>git submodule</code> command manages external dependencies within a Git repository. It allows you to include external repositories as submodules within your main repository. This is useful when you want to incorporate code from external sources while keeping it separate from your main project's codebase.</p>
<p dir="ltr" class="p-margin">To update a submodule in Git, you can use the following steps:</p>
<ol>
<li>Navigate to the directory of the submodule within your main repository.</li>
<li>Use <code>git fetch</code> to fetch the latest changes from the submodule's remote repository.</li>
<li>If you want to update to the latest commit on the branch tracked by the submodule, you can use <code>git pull</code>.</li>
<li>Alternatively, if you want to update to a specific commit or branch, you can use <code>git checkout</code> followed by the desired commit hash or branch name.</li>
<li>Once you have updated the submodule to the desired state, you need to commit the changes to the main repository to reflect the updated submodule state.</li>
</ol>
<h3 dir="ltr" id="20.-what-is-the-significance-of-git-push---force-with-lease-over-git-push---force?-the&lt;c">20. What is the significance of git push --force-with-lease over git push --force?</h3>
<p dir="ltr" class="p-margin">The <code>git push --force-with-lease</code> is a more cautious approach to force-pushing changes to a remote repository than <code>git push --force</code> because it prevents accidentally overwriting changes made by others on the remote repository.</p>
<p dir="ltr" class="p-margin">When you use <code>git push --force</code>, you force-push your changes to the remote repository regardless of whether others have updated it since your last fetch. This can lead to the unintentional loss of other developers' work.</p>
<p dir="ltr" class="p-margin">In contrast, <code>git push --force-with-lease</code> is a safer alternative. It checks whether the remote branch you are pushing to has been updated by others since your last fetch. If the remote branch has been updated, the push is rejected, preventing you from overwriting other developers' changes unintentionally.</p>
